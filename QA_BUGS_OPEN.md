**[P2] `checkin-form.tsx` — текст ошибки создания препарата не показывается пользователю**
Файл: `web/app/(app)/checkin/checkin-form.tsx` (`handleAddMed`, `addMedStatus`)
При ошибке `createMed()` (например, новая проверка "Такой препарат уже есть в списке") кнопка на экране чек-ина показывает только символ "!" на 2 секунды — сам текст ошибки нигде не рендерится. На `/habits/bulk` (`habit-manager.tsx`) тот же сценарий уже показывает `res.error` полным текстом — несогласованность между двумя формами добавления препарата.

**[P2] `DrugClassPrompt` может предложить классифицировать архивный препарат**
Файл: `web/app/(app)/checkin/meds/med-calendar.tsx` (`DrugClassPrompt`, вызывается с полным `meds` — включая архивные)
Фильтр `pending = meds.filter(m => m.isAsNeeded && m.drugClass === "unclassified")` не исключает `m.archived` — если архивируется ещё не классифицированный as-needed препарат, `/checkin/meds` продолжит предлагать классифицировать то, что пользователь уже удалила. Сейчас на живых данных не проявляется (все архивные уже классифицированы), но не защищено от будущего случая.
Рекомендация: добавить `&& !m.archived` в фильтр `pending`.

**[P2] `autoRegisterDetectedMeds` — теоретическая коллизия `id` между пользователями при одновременном импорте**
Файл: `web/app/(app)/import/migrebot/actions.ts`
`id: custom_${Date.now()}_${i}` уникален в рамках одного вызова (индекс `i` разруливает коллизии внутри цикла), но если два разных пользователя импортируют CSV в одну и ту же миллисекунду с одинаковым индексом обнаруженного препарата — возможна коллизия PRIMARY KEY. Вероятность крайне низкая (нужна миллисекундная одновременность), но раньше ошибка insert проглатывалась молча без обработки.
Статус: **частично пофикшено в этом раунде** — insert теперь проверяет `error` и логирует его вместо молчаливого проглатывания, так что редкая коллизия больше не обрушит остальной импорт незаметно. Сама возможность коллизии не устранена.
Рекомендация на будущее: перейти на `gen_random_uuid()` для `id`, если/когда таблица будет рефакториться под что-то, не завязанное на человекочитаемые id.

**[P2] Bottom sheet-модалки (в т.ч. новая "Другая боль") не учитывают `safe-area-inset-bottom`**
Файлы: `web/app/(app)/checkin/pain-quick-log-sheet.tsx`, `web/app/(app)/training/schedule/schedule-calendar.tsx`
Оба bottom sheet используют фиксированный `pb-6` вместо `env(safe-area-inset-bottom)` — на iPhone с home-indicator кнопка "Готово ✓"/сохранения может визуально прижиматься ближе к индикатору, чем остальной UI (который в `layout.tsx` использует `env(safe-area-inset-top)` для верха, но низа это не касается). Не регрессия нового раунда — новый компонент просто унаследовал уже существующий паттерн.
Рекомендация: добавить `paddingBottom: max(1.5rem, env(safe-area-inset-bottom))` в оба компонента одновременно, раз паттерн общий.

**[P2] `MIGREBOT_MED_PATTERNS` (ранее `MED_PATTERNS`) — эвристика по regex может давать ложные срабатывания**
Файл: `web/lib/migrebot-meds.ts` (переехал из `med-calendar.tsx` в общий файл в этом раунде — теперь используется и клиентским детектом, и серверным `autoRegisterDetectedMeds`)
Тот же риск, что и раньше: простые regex по подстроке (например `/спрей|назальн/i`) могут ошибочно засчитать приём препарата в свободнотекстовых заметках MigreBot. Теперь ложное срабатывание в тексте импорта может ещё и **создать несуществующий препарат** через `autoRegisterDetectedMeds` (раньше эффект ограничивался только неверной раскраской клиентского heatmap). Оригинальный текст по-прежнему показывается рядом (`MigreBotSection`), так что пользователь может заметить и разобраться вручную.
Рекомендация: не менять, если не появятся жалобы на конкретные ложные срабатывания; при появлении — рассмотреть явное подтверждение перед авто-регистрацией вместо тихого создания.

**[P2] `/checkin/meds` — нет явного пустого состояния, если ни один heatmap не проходит фильтр 0-intake**
