**[P0] `saveBulkHabits` не фильтрует по пользователю вообще**
Файл: `web/web/app/(app)/habits/bulk/actions.ts:21-47`
Проблема: Функция читает `daily_log` через `select("*").in("log_date", dates)` без какого-либо `app_user_id` фильтра, а затем делает `upsert(..., { onConflict: "log_date" })`. При этом миграция `021_user_isolation.sql` заменила уникальный констрейнт `daily_log_log_date_key` на `daily_log_user_date_key UNIQUE(app_user_id, log_date)` — констрейнта `log_date` больше не существует.

Рекомендация: Добавить `getAppUserId()`/`byUser()` в `select` и `upsert`, поставить `app_user_id` в каждую строку payload, исправить `onConflict` на `"app_user_id,log_date"`.

## P1 — Важные баги

**[P1] `redirect()` внутри `try/catch` — эффективно проглатывается**
Файл: `web/web/app/register/actions.ts:13-32`
Проблема: `redirect()` в Next.js бросает исключение `NEXT_REDIRECT`, которое ловится тем же `try/catch`, что оборачивает `createUser`. В частности, `redirect("/onboarding")` на строке 16 находится **внутри** `try` — если он выполнится успешно, это нормально (redirect должен пробросить исключение наружу из функции), но конструкция хрупкая: любое исключение из кода после `redirect()` (например, второй `createUser`/`setAuthCookie` внутри вложенного `try` на строках 22-29) тоже полагается на то, что redirect добрасывается через несколько уровней catch. Второй вложенный `try/catch` (строки 22-29) ловит **всё**, включая потенциальный `NEXT_REDIRECT` от `redirect("/onboarding")` на строке 25, и в catch-блоке просто делает ещё один `redirect("/register?e=2")` — то есть если после успешного `createUser`/`setAuthCookie` redirect на `/onboarding` по какой-то причине попадает в catch (например, Next меняет поведение или добавляется код между строками), пользователь молча не увидит /onboarding.
Воспроизведение: Пограничный сценарий "email уже существует, но онбординг не завершён" → `deleteUnfinishedUser` → повторный `createUser` → `setAuthCookie` → `redirect("/onboarding")` — весь этот блок обёрнут в `try` с `catch` который перехватывает `NEXT_REDIRECT`. Технически Next.js специально не считает `NEXT_REDIRECT` ошибкой в самом верхнем вызывающем коде, но вложенность try/catch здесь избыточна и рискованна — при рефакторинге (например, добавлении `await` кода после redirect) это сломается тихо.
Рекомендация: Не оборачивать `redirect()` в try/catch. Разделить: сначала выполнить всю бизнес-логику (create/delete/create) в try/catch без redirect внутри, затем вызывать `redirect()` один раз после успешного получения `user`, вне какого-либо catch.

**[ИСПРАВЛЕНО ✓]** Перепроверено в `web/app/register/actions.ts:1-39` (2026-07-01). Именно рекомендованная структура: `redirect()` вызывается ровно один раз, в самом конце функции, вне какого-либо `try/catch`. Внутри `try/catch` (и вложенного `catch`) теперь только присваивание переменной `target` (строка) — исключения из бизнес-логики (`createUser`/`setAuthCookie`/`deleteUnfinishedUser`) ловятся и превращаются в целевой путь, но сам `redirect(target)` вызывается уже после выхода из обоих try/catch блоков, поэтому `NEXT_REDIRECT` не может быть случайно проглочен. Единственная оставшаяся деталь (не баг): `redirect("/register?e=3")` на строке 11 (проверка длины пароля) — это ранний return-стиль redirect, он не в try/catch и корректен сам по себе. Логика полностью соответствует тому, что было рекомендовано.

**[P1] `getRecentWearableData` / `getRecentSleepData` считают "days ago" в UTC**
Файл: `lib/data.ts:449-458` и `491-499`
Проблема: `const since = new Date(); since.setDate(since.getDate() - days); ... since.toISOString().slice(0,10)`. `toISOString()` всегда возвращает UTC-дату. Для пользователя в часовом поясе UTC+3 (Россия) в промежутке 00:00–03:00 локального времени это даст дату "вчера" по UTC, сдвигая окно на день раньше, чем ожидается.
Воспроизведение: Между полночью и 3 утра по Москве окно "последние 7/28 дней" будет заканчиваться на день раньше, чем должно — минимальный, но систематический сдвиг данных wearable/sleep.

Но `getCurrentWeight` не наследует этот guard: раз `w.length === 0`, она безусловно возвращает `seed.WEIGHT.currentKg` = `66.75` — жёстко захардкоженный реальный вес Марины из `web/lib/seed-data.ts:35`, независимо от того, легаси это пользователь или совершенно новый аккаунт.
Воспроизведение: Новый пользователь регистрируется → проходит онбординг → на шаге 4 либо вводит свой вес (тогда после `saveWeight` в БД появится строка `weight_entry` и баг не проявится), либо жмёт «Пропустить» (кнопка `handleSkipWeight` в `web/app/onboarding/page.tsx:98-108` вызывает только `completeOnboarding()`, не создавая никакой `weight_entry`) → на `/dashboard` в карточке «вес» отображается `66.75` (плюс `weightDelta = currentWeight - weightStartKg`, где `weightStartKg` берётся из `user.weightStartKg` — для нового юзера это `null` из БД, значит fallback на `WEIGHT_GOAL.startKg = 74.2`, тоже сид Марины) → пользователь видит чужие персональные данные о весе на первом же экране после регистрации. То же самое произойдёт на `/checkin` (`weightPlaceholder={weight}` — плейсхолдер в форме будет 66.75) и на `/weight` (крупная цифра "66.75" как «вес сегодня», плюс `fromStart`/`toGoal` относительно сид-констант `WEIGHT_GOAL.startKg=74.2`/`WEIGHT_GOAL.kg=54`, а не относительно `user.weightGoalKg`/`weightStartKg`, которые для нового юзера уже могли быть сохранены на шаге 4 онбординга).
Предположение о причине: При рефакторинге под мультиюзер (`getRecentActualWeights` получила guard "новый юзер → пустой список, не сид Марины"), забыли применить тот же guard к `getCurrentWeight`, хотя она вызывает `getRecentActualWeights` и явно рассчитана на тот же сценарий "нет своих записей веса".
Рекомендация: В `getCurrentWeight` заменить безусловный `seed.WEIGHT.currentKg` на условный fallback (как в других функциях `data.ts`): если `uid` — новый (не legacy) и записей нет, вернуть `null`/`0`, а не сид Марины; вызывающему коду (`dashboard`, `checkin`, `weight`) явно обработать «веса ещё нет» (например, показать «—» вместо цифры). Дополнительно: `web/app/(app)/weight/page.tsx` вообще не вызывает `getCurrentUser()` и всегда считает `fromStart`/`toGoal` от `WEIGHT_GOAL.startKg`/`WEIGHT_GOAL.kg` (сид-константы) — нужно аналогично `dashboard/page.tsx:86-87` подставлять `user.weightGoalKg ?? WEIGHT_GOAL.kg` и `user.weightStartKg ?? WEIGHT_GOAL.startKg`.

**[P1] Длина цикла, введённая на шаге 2 онбординга, нигде не сохраняется и не используется**
Файл: `web/app/onboarding/actions.ts:19-28` (`saveCycle`), `web/app/onboarding/page.tsx:166-174` (UI шага 2)
Проблема: На шаге 2 онбординга пользователь вводит `cycleLen` (число от 21 до 45, дефолт 28) в поле «Длина цикла (дни)». `handleNext()` вызывает `saveCycle(lastPeriod, cycleLen)` (`page.tsx:78`). Но в `actions.ts`:
```ts

  if (!db) throw new Error("DB unavailable");

```
Параметр явно переименован в `_cycleLength` (со знаком подчёркивания — типичный маркер "намеренно не используется") и нигде не используется. Таблица `cycle_start` хранит только `start_date`, длина цикла нигде не персистится (ни в `app_user`, ни отдельной колонкой).
Воспроизведение: Пользователь с реальным циклом 32 дня вводит на онбординге "32". Значение никуда не пишется. `getCurrentCycle` (`web/lib/cycle.ts:80-96`) для пользователя с одной записью `cycle_start` вычисляет `stats = cycleStats([oneDate])` → `lengths = []` (нечего сравнивать, нужно ≥2 старта) → `recent = []` → `n = 1` (fallback) → `avg = 0/1 = 0` → `len = Math.round(0) || 28 = 28`. То есть независимо от того, что ввёл пользователь на шаге 2 (21–45 дней), движок цикла всегда стартует с дефолтных 28 дней, пока не накопится ≥2 реальных менструации. Предположение о причине: Похоже, `cycle_start`/`getCurrentCycle` изначально проектировались на "адаптивный" расчёт длины цикла из истории (комментарий в `cycle.ts:1`: «адаптивный прогноз, а не фиксированные 28 дней»), и разработчик сознательно решил не хранить ручной ввод длины — но тогда поле в UI онбординга вводит в заблуждение: обещает персонализацию, которой не будет, пока не накопится история.
Рекомендация: Либо (а) убрать поле «длина цикла» с шага 2 онбординга совсем (раз оно ни на что не влияет и движок всё равно всё вычисляет сам из истории), либо (б) реально использовать введённое значение как стартовый дефолт для `getCurrentCycle` до накопления истории — например, сохранить `cycle_length_default` в `app_user` и передавать его как fallback вместо жёстко зашитого `28` в `cycle.ts:83` (`const len = Math.round(stats.avgLength) || 28;` → `|| (user.cycleLengthDefault ?? 28)`).
