# QA Аудит ВЕРТА — 2026-07-01

Статический аудит кода (без запуска сервера, без обращения к БД). Ветка `feat/multiuser` (только что влит большой мультиюзер-рефакторинг). Проверены файлы из `web/web/lib/` и `web/web/app/(app)/{checkin,medical,training,habits,import,insights,dashboard}`, `web/web/app/login`, `web/web/app/register`, плюс сверка со схемой миграций в `web/supabase/migrations/`.

**Важный контекст безопасности**: в проекте включён RLS на таблицах, но единственная реально работающая политика (`010_balance_wheel.sql`) — это `using (true) with check (true)` для service_role. Приложение везде использует `supabaseAdmin()` (service_role), который **обходит RLS полностью**. Это значит, что изоляция пользователей на 100% держится на самодельном хелпере `byUser(query, uid)` в каждом запросе — если разработчик забыл его вызвать, БД не спасёт. Отсюда P0 находки ниже.

## Резюме

Общая оценка: мультиюзер-миграция сделана в целом аккуратно (паттерн `byUser`/`toUid` применён почти везде), но есть несколько мест, где фильтр по `app_user_id` пропущен полностью — это прямые межпользовательские утечки данных, а не гипотетические. ## P0 — Блокеры

**[P0] `saveBulkHabits` не фильтрует по пользователю вообще**
Файл: `web/web/app/(app)/habits/bulk/actions.ts:21-47`
Проблема: Функция читает `daily_log` через `select("*").in("log_date", dates)` без какого-либо `app_user_id` фильтра, а затем делает `upsert(..., { onConflict: "log_date" })`. При этом миграция `021_user_isolation.sql` заменила уникальный констрейнт `daily_log_log_date_key` на `daily_log_user_date_key UNIQUE(app_user_id, log_date)` — констрейнта `log_date` больше не существует.
Воспроизведение: (а) Любой пользователь, открывший `/habits/bulk`, при сохранении читает и подмешивает в апдейт **чужие** daily_log строки за те же даты (первая попавшаяся по `log_date`, независимо от владельца) — реальная утечка/порча данных между пользователями. (б) `onConflict: "log_date"` ссылается на несуществующий unique constraint → Supabase вернёт ошибку `there is no unique or exclusion constraint matching the ON CONFLICT specification` на каждый сохранение — фича, вероятно, полностью не работает после мультиюзер-миграции.
Рекомендация: Добавить `getAppUserId()`/`byUser()` в `select` и `upsert`, поставить `app_user_id` в каждую строку payload, исправить `onConflict` на `"app_user_id,log_date"`.

**[P0] `deleteMed` удаляет препарат без проверки владельца**
Файл: `web/web/app/(app)/checkin/actions.ts:99-107`
Проблема: `db.from("medication").delete().eq("id", id)` — нет `byUser`/`app_user_id` фильтра. При этом для сидовых препаратов id — общеизвестные строки (`sumatriptan`, `nurofen` и т.п.), одинаковые у всех пользователей.
Воспроизведение: Любой пользователь может передать в `deleteMed` id чужого препарата (в т.ч. предсказуемый seed-id) и удалить запись, которая физически принадлежит другому аккаунту (или общей seed-записи, которую видят/использует другой пользователь).
Рекомендация: Добавить `byUser(db.from("medication").delete().eq("id", id), uid)`.

**[P0] `getSignedUrl` не проверяет владельца файла**
Файл: `web/web/app/(app)/medical/actions.ts:127-132`
Проблема: Функция принимает произвольный `filePath: string` и выдаёт подписанный URL на 1 час без всякой проверки, что этот файл принадлежит текущему пользователю (`db.storage.from("medical-records").createSignedUrl(filePath, 3600)`).
Воспроизведение: `getSignedUrl` — экспортируемый server action (доступен клиенту как RPC). Зная (или подобрав/увидев в чужой сессии) путь вида `records/<uid>/<ts>_<filename>`, любой авторизованный пользователь получает подписанную ссылку на скачивание чужого медицинского документа. Сейчас функция вызывается только из `getRecords()` (которая уже отфильтрована по `byUser`), поэтому прямо сейчас эксплуатация требует вызова экспортированного action напрямую (через devtools/curl к RSC action endpoint) — но это именно тот случай, где нельзя полагаться на "просто из UI так не вызвать".
Рекомендация: Перед созданием signed URL проверить, что `filePath` начинается с `records/${uid}/` (или сделать выборку из `medical_record` по `file_path` + `byUser` и только затем подписывать).

## P1 — Важные баги

**[P1] `redirect()` внутри `try/catch` — эффективно проглатывается**
Файл: `web/web/app/register/actions.ts:13-32`
Проблема: `redirect()` в Next.js бросает исключение `NEXT_REDIRECT`, которое ловится тем же `try/catch`, что оборачивает `createUser`. В частности, `redirect("/onboarding")` на строке 16 находится **внутри** `try` — если он выполнится успешно, это нормально (redirect должен пробросить исключение наружу из функции), но конструкция хрупкая: любое исключение из кода после `redirect()` (например, второй `createUser`/`setAuthCookie` внутри вложенного `try` на строках 22-29) тоже полагается на то, что redirect добрасывается через несколько уровней catch. Второй вложенный `try/catch` (строки 22-29) ловит **всё**, включая потенциальный `NEXT_REDIRECT` от `redirect("/onboarding")` на строке 25, и в catch-блоке просто делает ещё один `redirect("/register?e=2")` — то есть если после успешного `createUser`/`setAuthCookie` redirect на `/onboarding` по какой-то причине попадает в catch (например, Next меняет поведение или добавляется код между строками), пользователь молча не увидит /onboarding.
Воспроизведение: Пограничный сценарий "email уже существует, но онбординг не завершён" → `deleteUnfinishedUser` → повторный `createUser` → `setAuthCookie` → `redirect("/onboarding")` — весь этот блок обёрнут в `try` с `catch` который перехватывает `NEXT_REDIRECT`. Технически Next.js специально не считает `NEXT_REDIRECT` ошибкой в самом верхнем вызывающем коде, но вложенность try/catch здесь избыточна и рискованна — при рефакторинге (например, добавлении `await` кода после redirect) это сломается тихо.
Рекомендация: Не оборачивать `redirect()` в try/catch. Разделить: сначала выполнить всю бизнес-логику (create/delete/create) в try/catch без redirect внутри, затем вызывать `redirect()` один раз после успешного получения `user`, вне какого-либо catch.

**[P1] `getMonthHabitStats` падает на месяцах короче 31 дня**
Файл: `lib/data.ts:319-333`
Проблема: `.gte("log_date", `${ym}-01`).lte("log_date", `${ym}-31`)` — для февраля/апреля/июня/сентября/ноября `${ym}-31` — невалидная дата (`2026-02-31`). PostgREST/Postgres кастует строковый литерал в тип `date` и упадёт с ошибкой на невалидной дате. Код не проверяет `error` (`const { data } = await byUser(...)`), поэтому ошибка проглатывается молча, `rows = []`, функция тихо возвращает `{ done: 0, daysLogged: 0 }`.
Воспроизведение: На дашборде для любого месяца с 28/29/30 днями (например, текущий июль 2026 не пострадает, а вот показатели за февраль или сентябрь всегда будут "0% за месяц"), статистика привычек за месяц будет занижена/обнулена без видимой ошибки.
Рекомендация: Использовать корректную верхнюю границу — первый день следующего месяца через `lt`, как уже сделано в `getTriptanCount` (строки 89-93 того же файла, где есть правильный паттерн nextY/nextM).

**[P1] `getRecentWearableData` / `getRecentSleepData` считают "days ago" в UTC**
Файл: `lib/data.ts:449-458` и `491-499`
Проблема: `const since = new Date(); since.setDate(since.getDate() - days); ... since.toISOString().slice(0,10)`. `toISOString()` всегда возвращает UTC-дату. Для пользователя в часовом поясе UTC+3 (Россия) в промежутке 00:00–03:00 локального времени это даст дату "вчера" по UTC, сдвигая окно на день раньше, чем ожидается.
Воспроизведение: Между полночью и 3 утра по Москве окно "последние 7/28 дней" будет заканчиваться на день раньше, чем должно — минимальный, но систематический сдвиг данных wearable/sleep.
Рекомендация: Вычислять `since` в локальной таймзоне пользователя (например, через `startOfDay`/локальные Y-M-D компоненты как в `lib/cycle.ts`), а не `toISOString()`.

**[P1] `deleteWorkout` / `bulkDeleteWorkouts` удаляют `workout_exercise` без проверки владельца**
Файл: `web/web/app/(app)/training/actions.ts:123-135` (`deleteWorkout`), `198-211` (`bulkDeleteWorkouts`)
Проблема: Обе функции сначала делают `db.from("workout_exercise").delete().eq("workout_id", id)` (или `.in("workout_id", ids)`) без всякой проверки владения, и только потом `workout_log` удаляется через `byUser`. Если `id` не принадлежит текущему пользователю, `workout_log` не удалится (RLS/byUser сработает), но связанные `workout_exercise` для **чужой** тренировки уже будут удалены до этой проверки.
Воспроизведение: Пользователь A вызывает `deleteWorkout(id)` с `id` чужой (пользователя B) тренировки. Строка `workout_log` пользователя B останется (фильтр `byUser` не даст её удалить), но её `workout_exercise` (все подходы/повторы/веса) будут стёрты, потому что удаление exercises не фильтруется по владельцу вообще.
Рекомендация: Сначала проверить владение через `byUser` на `workout_log` (или джойн), и только если подтверждено — удалять `workout_exercise`. Либо: выполнять удаление exercises только если `workout_log` delete подтвердил затронутую строку.

**[P1] `getWorkoutTemplates`/`getWeeklySchedule` помечены "global shared" в комментарии, но фильтруются `byUser`**
Файл: `lib/data.ts:398-427`
Проблема: Комментарий гласит `// Global shared templates — no per-user filtering`, но код тут же передаёт запрос через `byUser(..., uid)`, то есть реально фильтрует по `app_user_id`. Для пользователей без собственных шаблонов (`uid` — новый UUID, ни одной строки с этим `app_user_id`) `getWorkoutTemplates()` вернёт **пустой список**, а не общие сидовые шаблоны, потому что `byUser` для не-legacy uid делает `.eq("app_user_id", uid)`, что не совпадёт с NULL-строками сидовых шаблонов.
Воспроизведение: Новый зарегистрированный пользователь открывает `/training` → создаёт тренировку типа "Силовая" → `findTemplate` ищет по `templates` (полученным из `getWorkoutTemplates()`) → список пуст → упражнения не подставляются автоматически, хотя по замыслу (согласно комментарию) должны быть общими для всех.
Рекомендация: Либо действительно сделать выборку без фильтра пользователя (раз комментарий говорит "global"), либо, если объединение сид-шаблонов (`app_user_id IS NULL`) и пользовательских — задумано, использовать `.or(app_user_id.is.null,app_user_id.eq.${uid})` вместо `byUser`.

**[P1] Missing `router.refresh()` после `deleteMed`/`createMed` в UI — нужно перепроверить связанный компонент вручную**
Файл: `web/web/app/(app)/checkin/actions.ts:74-107` (проверить вызывающий компонент, не входящий в список аудита)
Проблема: Оба action используют `revalidatePath("/checkin")`/`revalidatePath("/dashboard")`, что инвалидирует серверный кеш, но если вызывающий клиентский компонент не делает `router.refresh()` после успешного ответа, список препаратов на экране может не обновиться сразу (Router Cache клиента живёт 30 сек по умолчанию).
Воспроизведение: Не проверено напрямую (компонент вне списка аудита), но паттерн risky — стоит проверить компонент, который рендерит список meds на `/checkin`, на предмет `router.refresh()` после `createMed`/`deleteMed`.
Рекомендация: Убедиться, что клиентский компонент со списком meds вызывает `router.refresh()` сразу после успешного `createMed`/`deleteMed` (по аналогии с `upload-form.tsx:78` и `bulk-form.tsx:82`, где это сделано правильно).

## P2 — Улучшения и edge cases

**[P2] `getAllHabits` двойной запрос при ошибке — неэффективно, но не баг**
Файл: `lib/data.ts:172-194`
Проблема: При ошибке первого запроса (например, отсутствие колонок `started_month`/`ended_month` на старой БД) делается второй запрос-фоллбэк. Рабочий код, но означает, что при систематической проблеме (не хватает колонки) каждый вызов делает 2 похода в БД.
Рекомендация: Если фоллбэк больше не нужен (миграции применены), убрать; если нужен — оставить с комментарием почему.

Файл: `web/web/app/(app)/import/migrebot/actions.ts:68-94`
Проблема: `previewCSV` не привязан к пользователю (это ОК, т.к. просто парсинг для UI), но стоит убедиться, что `previewCSV` не сохраняет ничего в БД по ошибке в будущем рефакторинге — сейчас безопасно, но нет теста/защиты.

**[P2] `MedCalendar`/`med-calendar.tsx` — эвристика распознавания медикаментов по regex может давать ложные срабатывания**
Файл: `web/web/app/(app)/checkin/meds/med-calendar.tsx:35-61`
Проблема: `MED_PATTERNS` использует простые regex по подстроке (например `/спрей|назальн/i`). Если в свободнотекстовом поле `meds` пользователь напишет что-то вроде "не помог спрей от насморка", это засчитается как приём `nasal_spray`, хотя мигренозного спрея не было.
Рекомендация: Не блокер (это UX-эвристика для отображения истории), но стоит задокументировать false-positive риск или показывать оригинальный текст рядом (что уже частично сделано через `MigreBotSection`).

**[P2] `toWeeks`/`buildDayRange` в `med-calendar.tsx` — потенциальный тяжёлый цикл на больших диапазонах**
Файл: `web/web/app/(app)/checkin/meds/med-calendar.tsx:8-19`
Проблема: `buildDayRange` генерирует посуточный массив (`while (cur <= to)`), а `addDays` каждый раз создаёт новый `Date` через `new Date(iso + "T12:00:00")`. Для 3-месячного окна (~90 дней) это не проблема, но если когда-нибудь диапазон расширят (например, "весь год"), будет 365 итераций с созданием Date объектов — не критично, но стоит иметь в виду при расширении фичи.
Рекомендация: Не менять сейчас, отметить как потенциальный узкий момент для будущего расширения диапазона.

**[P2] `Dashboard`: деление на ноль защищено, но `weights` может быть массивом из одного элемента**
Файл: `web/web/app/(app)/dashboard/page.tsx:90-99`
Проблема: `wMax - wMin || 1` защищает от деления на 0, если все веса одинаковые — это ок. Но `weights.length - 1 || 1` в знаменателе `x` даёт корректный fallback только для ровно одного элемента; если `weights` пуст (`getRecentActualWeights` теоретически может вернуть `[]` для нового пользователя без записей веса), `wMin`/`wMax` = `Math.min()`/`Math.max()` без аргументов = `Infinity`/`-Infinity`, что даст `NaN` в `spark`-строке точек polyline.
Воспроизведение: Новый пользователь без единой записи веса → `getRecentActualWeights` вернёт `[]` (см. `lib/data.ts:69` — `if (uid && uid !== "__legacy__") return []`) → `ws = []` → `Math.min(...[])` = `Infinity` → `spark` содержит `NaN,NaN ...` → SVG polyline рисуется пустым/некорректным (не крашится, но визуально сломанный график).
Рекомендация: Добавить guard: если `weights.length === 0`, не рендерить sparkline или показывать заглушку.

**[P2] `sport_type` и `medication` global fallback наборы (`DEFAULT_SPORT_TYPES`, seed meds) — задокументированное поведение, но стоит явно тестировать онбординг нового пользователя**
